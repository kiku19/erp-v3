Overview

Executive Summary

OBS and Resource Pool are the foundation layer — every other module depends on them. Without OBS, resources have no organisational home. Without Resource Pool, activities have no one to assign. These two are combined into a single phase because they are tightly coupled and cannot deliver independently.

OBS gives you
Structure
Departments, roles, hierarchies, rate cards, approval chains, cost centre mapping
Resource Pool gives you
People & Assets
Persons, equipment, materials — populated into the OBS structure
Together they give you
Living Workforce Model
Scoped resource pickers, auto-rates, approval routing, cross-project conflict detection
Mental model: OBS is the empty org chart. Resource Pool is filling that chart with real people and assets. One without the other is useless.

1.3 — Rate Card Structure

Rate cards live on OBS roles and are the source of truth for all cost planning. When a resource is assigned to an activity, the rate is pulled from here — not typed by the PM.

Past activity costs use the rate card valid at the time of execution.

HRM Integration

Ownership Boundary

The most important design decision. HRM and OPUS must never fight over the same data. Every field must have one clear owner.

HRM Owns (Source of Truth)
Employee master — name, ID, join date, employment type
Org hierarchy → maps to OBS nodes
Compensation — salary, pay type
Leave balances and approved leaves
Certifications and training records
Attendance / check-in check-out records
Exits and deactivations
OPUS Owns (Project Layer on Top)
Productivity rates per activity type
Project allocation calendar
Activity-level timesheet entries
Skill tags relevant to project work
Rate overrides for specific projects
Equipment records (HRM never touches these)
Material catalogue (OPUS + procurement)
Conflict resolution rule: If HRM and OPUS data conflict — HRM is master for people data. OPUS is master for project data. Never let OPUS overwrite HRM records.
HRM Integration

Integration Scenarios

Scenario A
Full HRM Integration — Real-time API Sync
Best option when HRM system has an API. OPUS is a consumer of HRM data, not a recorder.
Event in HRMEffect in OPUS
New employee addedAuto-appears in OPUS Resource Pool. OBS node mapped from HRM department code.
Employee exitsPerson deactivated in OPUS. Past assignments retained for history. Cannot be assigned to new activities.
Org change (Ravi moves Civil → Electrical)OBS node updated. Future assignments route to Electrical. Past assignments retain Civil node for historical reporting.
Leave approved in HRMOPUS marks person as unavailable on those dates. If allocated to an activity — PM receives alert.
Attendance logged (biometric/app)OPUS pre-fills timesheet for that day. Worker only confirms activity — does not re-enter check-in/out.
Salary revision in HRMDaily allocation updated in OPUS. New rate applies to future activities. Past activities retain old rate.
Scenario B
Partial Integration — Periodic CSV/Excel Import
When HRM system has no API. Manual export-import cycle.
  • HR exports employee list as CSV/Excel periodically (weekly or monthly)
  • OPUS imports and reconciles — new employees added, exits deactivated, changes updated
  • Leave data: manual entry in OPUS availability calendar (HR sends approved leave report)
  • Attendance: fully manual — workers submit timesheets without pre-fill
  • Salary changes: imported in the same periodic cycle
Scenario C
Standalone — No HRM System
OPUS Resource Pool is the system of record. Designed for this from day one — HRM integration is an upgrade later.
  • Admin creates person records manually in OPUS Resource Pool
  • Pay type, rate, and role configured directly in OPUS
  • Leave managed within OPUS availability calendar
  • Timesheet is the only attendance record
  • HRM fields are clearly tagged internally so integration can replace them cleanly later

HRM Integration

The Key Decision

Before building Phase 1, decide: Is OPUS the system of record for people data — or is OPUS a consumer of HRM data?
A
OPUS is system of record

Build full person record management in OPUS. HRM integration is additive later.

  • Works on day one — no HRM dependency
  • More fields to maintain manually
  • HRM integration replaces/locks fields when added
  • Best for companies without an existing HRM
B
OPUS consumes HRM data

Build a thin person record in OPUS — project-relevant fields only. HRM holds the rest.

  • Integration is a hard dependency before go-live
  • Less maintenance in OPUS long-term
  • Best for companies with existing HRM (SAP, Darwinbox, etc.)
  • OPUS-specific fields always stay in OPUS regardless
Recommendation: Build OPUS as system of record first (Scenario C — standalone). Tag every field that would be owned by HRM so integration can cleanly replace it later. This way OPUS ships without waiting for HRM, and HRM becomes a clean upgrade — not a rebuild.
