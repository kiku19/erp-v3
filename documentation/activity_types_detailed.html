Fundamentals — What is an Activity?

The building block of every project plan in OPUS ERP

What an Activity Contains

Every activity in OPUS carries three core dimensions that together allow us to measure whether the work is done, how much it should cost, and whether we are on track.

Quantity
The Deliverable
e.g., 1,000 m² of wall, 500 bags of cement, 80 man-hours of code
Total Work Hours
The Effort
How many man-hours are planned for this activity
Cost / UOM
The Unit Cost
Cost per unit of measure → drives Budget at Completion (BAC)
How "Done or Not" is Measured

OPUS supports four methods to measure completion. The right method depends on the nature of the work:

MethodFormulaBest For
Physical QuantityActual Qty / Planned Qty × 100Field work, construction, manufacturing
Work HoursActual Hours / Planned Hours × 100Labor-driven tasks
Milestone / StepsSum of completed milestone weightsDesign, approval, document tasks
Manual EntryPM directly enters %Ambiguous tasks, override
Activity Status Logic
0% → Not Started
·
1–99% → In Progress
·
100% → Completed
·
% < 100% AND Planned Finish < Today → Overdue

To finish 305 m² remaining in 1 day, Ravi alone can only do ~100 m². The PM decides to crash the resources — adding 3 extra painters.

Resource Dependent Labor-intensive work

Resource Dependent Activity

Core rule: Duration is a computed output, not a stored value. It is always derived as Total Work ÷ Daily Resource Capacity. Add resources → system shortens duration. Remove resources → system extends it automatically.
Example — Paint Interior Rooms (Resource Dependent)

Activity Setup

TypeResource Dependent
Total Work80 man-hours (fixed)
ResourcesRavi + Suresh
Daily capacity8 + 8 = 16 hrs/day
Planned Duration80 ÷ 16 = 5 days
Key insight: Duration is not entered — it is calculated. If you change resources, OPUS recalculates the finish date instantly. The PM never types a duration here.
Duration = Total Planned Hours / Daily Resource Capacity = 80 hrs / 16 hrs/day = 5 days
Scenario — Suresh is Absent on Day 4

Same absence as before — but watch how Resource Dependent behaves differently from Task Dependent.

DayCapacityHours DoneCumulative HoursRemaining
Day 116 hrs161664 hrs
Day 216 hrs163248 hrs
Day 316 hrs164832 hrs
Day 48 hrs (Suresh absent)85624 hrs
Remaining after Day 4 = 80 − 56 = 24 hrs Ravi alone = 8 hrs/day → Days needed = 24 / 8 = 3 more days New Finish = Day 4 + 3 = Day 7 (was Day 5) → Duration auto-extends from 5 days to 7 days
The system recalculates the finish date automatically. No PM intervention needed to reflect reality. The Gantt bar stretches on its own.
Recovery Options for Resource Dependent
A
Accept the extension

Duration extends from 5 to 7 days. Cost stays the same — still 80 total hours at same rates. No extra spend.

Choose when the deadline has float or buffer. Client impact is manageable.
B
Add resource to recover

Add Painter 3 + 4 from Day 5. New capacity = 8 + 8 + 8 = 24 hrs/day. Remaining 24 hrs ÷ 24 = 1 day. Duration recovers to 5 days.

Choose when deadline is hard. System recalculates finish date backward to Day 5 automatically.
04

The Key Difference — Task vs Resource Dependent

This is the most important concept to understand before building or using the planning module.

Task Dependent

Duration is a fixed input entered by the PM. Resources work within that window. The work must fit the time.

  • Adding resources → duration unchanged
  • Resource absent → SV goes negative
  • Behind schedule → PM must manually crash or extend
  • Gantt bar is fixed width, progress fill moves inside
Resource Dependent

Duration is a computed output — never stored. Resources are the input. The time fits the work.

  • Adding resources → finish date moves earlier
  • Resource absent → finish date auto-extends
  • Behind schedule → system auto-reflects the slip
  • Gantt bar stretches or shrinks as resources change
It's not just a UI lock. The difference is architectural. In Resource Dependent, duration doesn't exist as a field — it is always a live calculation. You can't "unlock" it. In Task Dependent, duration is a stored value that the PM owns.

Same Scenario, Different Outcome

EventTask Dependent BehaviorResource Dependent Behavior
Suresh absent Day 4SV = −₹5,250. Duration still 5 days. Alert: behind schedule.Finish date moves from Day 5 → Day 7 automatically.
PM adds 3 painters Day 5Hours increase, more work done, duration holds at 5.System recalculates: finish moves back to Day 5.
No recovery actionActivity overdue. Cascade to successors.Finish simply slips. No manual correction needed.
05
Level of Effort (LOE) PM overhead, supervision, support

Level of Effort (LOE)

Core rule: LOE does not produce anything measurable. It exists as long as the activities it governs exist. Duration is borrowed. % complete is purely time-based and automatic. No quantity. No deliverable. Only continuous presence and cost accumulation.
How LOE Duration is Derived
LOE Start = min (Earliest Start of all governed activities) LOE Finish = max (Latest Finish of all governed activities) LOE Duration = auto-calculated between these two dates
The PM never enters a start date, end date, or duration for an LOE activity. These are always computed from the governed activity list.
Example — Office Renovation Project
ActivityPlanned StartPlanned FinishCalendar
A — Demolition workDay 1Day 56-day (Mon–Sat)
B — Electrical wiringDay 3Day 105-day (Mon–Fri)
C — PaintingDay 8Day 156-day (Mon–Sat)
D — Final inspectionDay 14Day 165-day (Mon–Fri)
LOE — Site SupervisorDay 1 (from A)Day 16 (from D)Own calendar
The supervisor governs all four activities. LOE automatically spans Day 1 → Day 16 = 16 working days. As activities A, B, C, D change, the LOE bar stretches or shrinks with them.
How % Complete is Measured in LOE
% Complete = Working Days Elapsed / Total Working Days × 100

At Day 8 of 16 working days:

% Complete = 8 / 16 × 100 = 50% ← advances automatically with the calendar
Nobody logs progress. Nobody enters quantities. The clock advances % complete on its own every working day.
Why LOE Has Its Own Calendar (Not Inherited)

Activities A and C use a 6-day calendar. Activities B and D use a 5-day calendar. Which does the LOE inherit?

Neither. Inheriting from governed activities doesn't work — it cannot decide which calendar to follow when they conflict.

The answer: LOE uses its own calendar, which defaults to the assigned resource's calendar.

What governed activities control
Span (boundaries)
Start = Day 1, Finish = Day 16. That's all they contribute to LOE.
What LOE's own calendar controls
Heartbeat
How many working days tick within that span. Drives cost and % complete.
Ex
Calendar Impact Example
LOE CalendarWorking days in spanSupervisor cost% at Day 8
6-day (resource calendar)17 working days17 × ₹500 = ₹8,5009/17 = 52.9%
5-day (wrong calendar)14 working days14 × ₹500 = ₹7,0008/14 = 57.1%
Wrong calendar = wrong cost and wrong % complete. This is why the calendar must be explicitly set and defaults to the resource's calendar.
What Happens When a Governed Activity Slips

Activity C (Painting) slips from Day 15 to Day 18 due to delayed materials.

Before slip
LOE: Day 1 – Day 16
Supervisor cost = 16 working days × ₹500 = ₹8,000
After Activity C slips
LOE: Day 1 – Day 18
Supervisor cost = 18 working days × ₹500 = ₹9,000
Schedule slippage automatically creates cost overrun in LOE. The supervisor works 2 extra days = ₹1,000 extra AC. This is how PM overhead cost gets captured honestly in the project EAC.
LOE and EVM — Critical Rule

LOE always reports exactly on schedule because % complete = time elapsed. This creates a misleading picture if included in EVM:

At any point in time for LOE: EV = PV (always — because % complete = time elapsed) SV = EV − PV = 0 (always zero — misleadingly "on schedule") CV = EV − AC (only this matters for LOE)
LOE must be excluded from project-level SV and SPI. Including it inflates EV and makes the project look healthier than it is. LOE only contributes to total project Actual Cost (AC) and EAC.

Milestone Zero duration — marks events, not work

Milestone Activities

Core rule: Milestones represent the beginning or end of a significant event. They have zero duration, no resources, and no quantity. They are purely binary — either achieved (100%) or not (0%).
Two Types of Milestones
Start Milestone

Marks the beginning of a phase or event. Has only a start date.

  • Site Handover Received
  • Client Kickoff Completed
  • Materials Delivered
  • Design Approval Received
Finish Milestone

Marks the completion of a phase or deliverable. Has only a finish date.

  • Foundation Work Complete
  • Factory Acceptance Test Passed
  • System Go Live
  • Client Handover Signed Off
Milestone Rules in OPUS
PropertyBehavior
DurationZero — always
ResourcesNone assigned
Quantity / CostNot applicable — hidden in UI
% CompleteBinary: 0% (not achieved) or 100% (achieved)
Contractual flagCan be flagged as contractual for client reporting
On GanttShown as a diamond ◆ not a bar

Every cost that hits an activity falls into one of four buckets. Each type has a different cost accumulation mechanism.

The Four Resource Types
TypeWhat it isCost DriverExample

MaterialConsumables — quantity basedQty consumed × Unit costPaint (litres), Thinner (litres)

How Each Type Accumulates into AC Differently
The critical distinction — they don't all tick the same way:
Labour
Ticks with TIME
Hours logged in timesheet each day
Material
Ticks with CONSUMPTION
Litres / kg / bags used logged in progress entry
Equipment
Ticks with USAGE / DEPLOYMENT
Days or hours the equipment is on site
Expense
Ticks at a POINT
One-time charge logged when incurred

Resource TypeAffects CV?Affects SV?How

Material Wastage Example — Day 3

Painters use 28 litres of paint instead of planned 20 litres due to rough surface and rework.

Planned paint cost Day 3 = 20L × ₹300 = ₹6,000 Actual paint cost Day 3 = 28L × ₹300 = ₹8,400 Material overrun = ₹2,400 EV (60% × ₹52,500) = ₹31,500 ← unchanged, wall doesn't know about wastage AC (cumulative with wastage) = ₹34,250 CV = ₹31,500 − ₹34,250 = −₹2,750 ← Over budget SV = ₹0 ← Still on schedule

Equipment Missing — Ladder not available Day 2

Painters can only reach the lower 4 feet of the wall. Upper portion inaccessible.

Planned output Day 2 = 200 m² Actual output Day 2 = 80 m² (lower portion only) Cumulative = 280 m² (was planned 400 m²) SV = (28% × ₹52,500) − (40% × ₹52,500) = ₹14,700 − ₹21,000 = −₹6,300 ← Behind
Equipment billing type matters here. If Ladder is daily rental → AC still ticks ₹500 even though it wasn't there (you're paying for equipment that blocked your progress). If pay-per-use → ₹0 that day. OPUS tracks this at the equipment resource level via Billing Type.

CV and SV Variance Attribution — Drill-Down Logic

When CV or SV goes negative, OPUS should tell the PM exactly why:

CV is negative — root cause drill-down
  • Labour overrun → someone logged excess hours vs planned
  • Material wastage → consumption rate exceeds baseline rate
  • Equipment idle → deployed but not productive (daily rental still ticking)
  • Expense surprise → unplanned expense hit
SV is negative — root cause drill-down
  • Labour absent → fewer hours → less quantity done
  • Material shortage → ran out mid-day → work stopped
  • Equipment missing → couldn't access area → output reduced
  • Rework → quantity counted, then redone → net zero output

Labour — Pay Type, Overtime & Productivity Rate

The basic labour model only tracks cost per hour. This section adds three dimensions that turn a timesheet entry into a workforce intelligence model: Pay Type (how cost is computed), Overtime flag (whether extra hours cost money), and Productivity Rate (how much output the person produces per hour).
The Problem with Hourly-Only for Salaried Workers

Ravi is a salaried employee at ₹52,000/month. He works 10 hours on Day 3 (2 hrs overtime).

Hourly model (wrong for salaried)
Daily allocation = ₹52,000 / 26 days = ₹2,000/day Hourly equivalent = ₹2,000 / 8 = ₹250/hr AC Day 3 = 10 hrs × ₹250 = ₹2,500 ← overstates cost He was not paid extra for overtime
Salaried model (correct)
AC Day 3 = ₹2,000 flat ← daily allocation only Overtime 2 hrs = tracked, not paid Cost is accurate. Hours are still visible.
The overtime hours are real work that happened. They should be tracked — for productivity analysis, future planning, and recognising when a person is consistently overloaded. But they should not inflate AC.
Pay Type — The New Field
Pay TypeHow AC is ComputedOvertimeRate Field
HourlyHours logged × hourly ratePaid — at overtime rate₹/hour
SalariedFixed daily allocation regardless of hoursTracked only — not paid (locked)₹/month or ₹/day
ContractFixed lump sum for full activityNot applicableFixed amount

How Pay Type Changes the Fields in OPUS

If Pay Type = HOURLY: Cost Rate = ₹/hour (mandatory) Overtime Rate = ₹/hour (mandatory if overtime allowed) AC = Regular hours × rate + Overtime hours × OT rate If Pay Type = SALARIED: Cost Rate = ₹/month → auto-converts to daily allocation Overtime flag = TRACKED ONLY (locked — no overtime pay field shown) AC = Working days allocated × daily rate (overtime hours logged but not costed) If Pay Type = CONTRACT: Cost Rate = Fixed lump sum for activity Overtime = Not applicable — hidden AC = Fixed amount logged on completion
Productivity Rate — The Key New Concept

Instead of only knowing cost per hour, OPUS should also know output per hour. This is the productivity rate.

Productivity Rate = Quantity produced / Hour worked = m²/hr (for painting) = bags/hr (for concrete laying) = units/hr (for assembly)

Three Ways to Set the Productivity Rate

1
Historical (auto-suggested)

OPUS looks at past activities for this person + same activity type and computes the average.

Ravi — past painting jobs: Activity A: 300m² / 24hrs = 12.5 m²/hr Activity B: 220m² / 20hrs = 11.0 m²/hr Activity C: 180m² / 15hrs = 12.0 m²/hr Suggested = 11.8 m²/hr (average)

OPUS pre-fills this when Ravi is assigned to a new painting activity.

2
Manual Override

PM looks at site conditions — rough surface, scaffolding needed — and decides Ravi will be slower than usual.

Historical: 11.8 m²/hr Manual override: 10 m²/hr Reason: rough exterior surface, scaffolding

PM judgment overrides historical average. Override reason is logged.

3
Computed Actual

During execution, OPUS computes the actual rate daily from quantity and hours logged.

Day 3: 100m² / 8hrs = 12.5 m²/hr → above planned 10 m²/hr ✓ This feeds back into Ravi's history for the next activity assignment.
How Productivity Rate Validates the Plan Before Execution

Once productivity rates are set, OPUS can check whether the plan is actually feasible before the baseline is frozen.

Planned Quantity: 1,000 m² Ravi's productivity: 10 m²/hr (manual override) Suresh's productivity: 8 m²/hr (historical average) Combined output: 18 m²/hr Hours needed = 1,000 / 18 = 55.6 hours Available hrs = 2 people × 8 hrs × 5 days = 80 hrs Check: 55.6 hrs < 80 hrs ✓ Plan is feasible — has buffer

What if the Plan is Not Feasible?

If only Ravi is assigned (10 m²/hr): Hours needed = 1,000 / 10 = 100 hrs Available = 1 × 8 × 5 = 40 hrs Gap = 60 hrs OPUS alert: "Insufficient capacity. At planned productivity, activity will complete only 400m² of 1,000m² in 5 days. Increase resources or extend duration to 13 days."
This pre-validates the plan before the baseline is frozen. No ERP tool does this by default — it is a significant differentiator for OPUS.
Predictive Warning During Execution

Productivity rate also enables OPUS to issue leading indicator alerts — warnings before SV even goes negative.

Day 2 — Ravi's actual productivity = 8 m²/hr (planned 10 m²/hr) OPUS computes trajectory: Remaining days = 3 Ravi + Suresh capacity = (8 + 8) × 8 hrs × 3 days = 192 m² Current completed = 200 m² (Day 1 + Day 2) Projected total = 200 + 192 = 392 m² Gap = 1,000 − 392 = 608 m² short OPUS alert (Day 2, before SV is negative): "At current productivity, activity will complete 39% of planned quantity by Day 5. Consider adding resources or reviewing work method."
SV hasn't gone negative yet — it's only Day 2. But OPUS can see the trajectory before the damage is done. This is far more valuable than discovering the problem on Day 5 when it is too late to recover without crashing.
Overtime Tracking for Salaried Workers — Day 3 Example

Ravi is salaried. Overtime pay flag = No. He works 10 hours on Day 3.

Overtime gave extra output. The productivity rate is captured correctly using total hours — 10 hrs, not 8 hrs.

What Consistent Overtime Tells You

If Ravi logs overtime consistently across 3 or more activities, OPUS should flag: "Resource regularly over-allocated — review planning assumptions." This signals that either activity durations are too short, or Ravi's planned hours are being systematically underestimated. Future planning should use a higher hour estimate for activities he is assigned to.
